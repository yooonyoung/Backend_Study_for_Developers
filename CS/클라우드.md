### 마이크로서비스 아키텍처(MSA)란 무엇인가요?

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>
    <br/>
    MSA는 어플리케이션의 단위를 '목적'으로 나누는 것입니다.<br/>
    모든 기능을 하나의 어플리케이션에서 비즈니스 로직을 구성해 운영하는 Monolithic 아키텍처와 달리, 기능(목적)별로 컴포넌트를 나누고 조합할 수 있도록 구축합니다. MSA에서 각 컴포넌트는 API를 통해 다른 서비스와 통신하는데, 모든 서비스는 각각 독립된 서버로 운영하고 배포하기 때문에 서로 의존성이 없습니다. 따라서 하나의 서비스에 문제가 생겨도 다른 서비스에 영향을 끼치지 않습니다.
</details>

### 하이브리드 클라우드를 주목하는 이유

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>
    <br/>
퍼블릭 클라우드와 프라이빗/온프레미스의 장점을 결합하여 비즈니스에 최적화된 인프라를 구성할 수 있기 때문입니다. <br/><br/>
퍼블릭 클라우드는 기업 외부에서 클라우드 사업자가 제공하는 서비스를 이용하는 것으로 자체 인프라를 구축하지 않아도 되는 장점이 있습니다.<br/>
프라이빗 클라우드는 기업에서 직접 관리하는 클라우드 시스템 또는 외부 데이터센터에서도 사설망으로만 접근할 수 있는 시스템을 말합니다. 직접 인프라를 구축해야 해서 초기 구축 비용과 각종 자원 사용료를 모두 부담해야 하지만 기업 특성에 맞게 운영할 수 있고 보안에 뛰어나다는 장점이 있습니다.<br/><br/>
따라서 기업은 신규 서비스 출시와 같은 예측할 수 없는 워크로드나 대용량 데이터 처리는 비교적 경제적이고 신속하게 자원을 확장할 수 있는 퍼블릭 클라우드를 이용하고, 핵심 비즈니스 또는 기밀이 중요한 데이터 관리와 백업은 프라이빗 클라우드나 온프레미스로 관리합니다. (확장성과 안정성 모두 충족)<br/><br/>
또, 온프레미스와 퍼블릭 클라우드를 병행 사용하면서 워크로드를 클라우드 환경으로 안전하게 이전할 수 있기 때문입니다. 인프라 시스템을 클라우드화하는 기업들은 퍼블릭 클라우드와 기존 온프레미스를 함께 사용하는 단계를 거칩니다.<br/>
클라우드로 전환하면서 가장 우려되는 보안, 개인정보, 예기치 못한 사고에 대비해 핵심 비즈니스는 온프레미스에 남겨두고, 추가적인 컴퓨팅 자원이 필요할 때 퍼블릭 클라우드를 이용해 바로 리소스를 증설해 사용하는 형태입니다.
</details>

### 컨테이너란?

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>
    <br/>
컨테이너는 보통 LXC(Linux Container)입니다. LXC는 호스트 OS에서 프로세스 간 벽을 만드는 기술로, 네임 스페이스와 cgroup을 조합한 형태입니다. 즉, 격리된 고유 영역에서 애플리케이션을 실행하는 것을 의미합니다. <br/><br/>컨테이너는 애플리케이션의 실행에 필요한 라이브러리와 바이너리, 기타 구성 파일을 '이미지' 단위로 빌드하여 패키지로 배포합니다. 따라서 실행에 필요한 모든 환경이 준비되어 있기 때문에 어떤 환경에서도 애플리케이션을 오류 없이 동작시킬 수 있습니다. OS가 포함되지 않아 크기가 작아 컨테이너 복제와 배포에 용이하며, 운영체제 부팅이 필요 없어 서비스 시작 시간도 짧습니다. 
</details>

### 롤링 업데이트 (Rolling Update)란

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>
    <br/>
롤링 업데이트는 일반적인 배포를 의미합니다. 단순하게 한 대씩 재시작하는 방식으로,(새로운 Replicaset을 만들고, 이전 Replicaset을 줄이면서 새로운 Replicaset의 수를 점차 늘려간다) 코드 변경에 따른 side effect가 발생할 수 있습니다. 하지만 롤백이 가능하고, 관리가 편해 많이 사용되는 방식입니다. 새로운 버전의 업데이트를 중단없이 사용자들에게 제공할 수 있습니다.
</details>

### 블루-그린(Blue-Green)

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>
    <br/>
예전 배포물을 블루(blue), 신규 배포물을 그린(green)이라고 해서 붙여진 이름입니다. 라우팅 바꿔치기 배포라고도 불리는데, 새로운 배포물을 배포하고 모든 연결을 새로운 배포물만 보게 합니다. 임시 도메인으로 green 어플리케이션에 배포하고, 실제 도메인을 green 앱에도 배포한 후 blue 앱에 실제 도메인을 제거합니다. 이후 임시 도메인을 green에서 제거합니다. 코드 변경에 따른 side effect가 없지만, 장애가 발생하면 크게 영향을 주게 됩니다. 이 방식은 쿠버네티스에서 두 가지의 별도 deployment가 필요하기 때문에 클러스터에 충분한 리소스가 있는지 확인하고 사용해야 합니다.
</details>

### 카나리(Canary)

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>
    <br/>
1대 또는 특정 user에게만 미리 배포했다가 잘 되면 전체에 배포하는 방식입니다. 수정한 코드가 워낙 많이 바뀌어 불안할 때 사용하는 방식입니다. 즉, 현재 서비스 중인 버전을 대다수 유저가 접근하도록 하고, 새로운 버전을 소수의 인원만 접속하게 합니다. 그래서 문제점을 더 빨리 찾아내고, 결과적으로 모든 user들이 새로운 버전으로 라우팅하게 됩니다. 문제가 발생하면 언제든지 구버전으로 롤백이 가능합니다.
</details>

### 클라우드가 유리한 경우

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>

### 온프레미스가 유리한 경우

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>
    <br/>
잠시라도 끊어져서는 안 되는 시스템이나 클라우드 업체가 보장하는 것 이상의 가용성이 필요한 시스템에서는 온프레미스가 유리합니다. 온프레미스는 서버가 죽지 않는 것을 목표로 하는 반면, 클라우드는 많은 인스턴스로 이루어진 분산 환경에서 인스턴스가 죽으면 다른 인스턴스로 빠르게 대체되는 것을 목표로 하기 때문입니다. 
또 기밀성이 높은 데이터나, 물리적인 저장 장소를 명확히 알 필요가 있을 때는 온프레미스가 유리합니다.<br/><br/>
- 유동적이지 않은 비즈니스 환경<br/>
- 변화가 많지 않고 안정적인 워크로드와 시스템<br/>
- X86을 기반으로 하지 않거나 리눅스 및 윈도우 이외에 다른 운영 체제에서만 작동되는 어플리케이션<br/>
- X86을 기반으로 하되, 가상화 환경에서 운영하기 어려운 어플리케이션<br/>
- 어플리케이션 마이그레이션을 실행할 개발 인력이 부족한 경우<br/>
- 이미 효율적이고 안정적으로 운영되고 있는 어플리케이션<br/>
- 물리 서버 내에서만 운영될 수 있는 어플리케이션
</details>

### CI/CD란?

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>
    <br/>
지속적 통합(CI: Continuous Integration), 지속적 전달(CD: Continuous Delivery), 지속적 배포(CD: Continuous Deployment)를 말합니다. 작은 단위의 코드를 짧고 빈번하게 통합하고, 테스팅을 위해 결과물을 짧고 빈번하게 전달하며, 프로덕션 환경까지 빠르게, 자동으로 배포하는 방식을 말합니다. CI/CD를 구성하는 이유는 급변하는 경영환경에 빠르게 대응하며 효율을 극대화하기 위해서이고, 반복되는 일을 자동화로 해결하기 위해서입니다. 
</details>

### HTTP vs HTTPS

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>
    <br/>
모두 서버/클라이언트에서 데이터를 주고받기 위한 프로토콜입니다. HTTP와 HTTPS의 다른 점은, HTTPS는 HTTP에 데이터 암호화가 추가되었다는 점입니다. HTTPS는 중간에 제3자가 정보를 볼 수 없도록 공개키 암호화를 지원하고 있습니다. HTTP는 80번 포트를 사용하고, HTTPS는 433번 포트를 사용합니다. 
</details>

### 4차 산업혁명 vs 디지털 트랜스포메이션

<details>
   <summary> 예비 답안 보기 (👈 Click)</summary>
    <br/>
디지털 트랜스포메이션은 디지털 기반으로 기업의 전략, 프로세스, 시스템 등을 근본적으로 변화시키는 경영 전략입니다. 4차 산업혁명은 기술적 변화에 따른 경제, 산업, 사회의 총체적 변화에 초점을 두고 있고, 디지털 트랜스포메이션은 기업의 경영 전략적 관점에서의 조직과 프로세스 등의 근본적 변화에 중점을 두고 있습니다.
</details>
